# Web: 定时任务灵动岛提示配置 - 技术方案

## 背景与目标

目前 `web-ui` 已具备两块基础能力：

1. 定时任务管理页面已完成创建、编辑、暂停、恢复、手动触发、日志查看。
2. Electron 端已具备灵动岛窗口、系统通知、托盘高亮和基础通知事件接收能力。

但两者还没有真正打通：

- 定时任务表单中没有“提醒策略”配置入口。
- 灵动岛通知数据结构是通用型的，缺少定时任务上下文。
- 当前灵动岛点击后只能关闭，无法稳定回到对应任务详情。
- Electron 本地配置只有全局 `notificationLevel`，没有“定时任务提醒偏好”的明确分层。

你现在已经在 Go 侧完善了定时任务相关逻辑，因此 `web-ui` 需要补齐“前端配置 + Electron 展示 + 交互回跳”这一层，使用户可以为定时任务配置是否通过灵动岛进行提示。

本方案目标：

- 在定时任务创建/编辑流程中增加提醒配置。
- 让 Electron 灵动岛能识别“来自定时任务”的事件并按配置展示。
- 让用户点击灵动岛后可以回到对应定时任务详情。
- 保持 `web-ui` 的边界：只做展示和配置，不承接定时业务判断。

## 现状分析

### 已有能力

#### 1. 定时任务页面

`src/views/Timer/TimerView.vue` 已支持：

- 定时任务列表
- 创建 / 编辑 / 删除
- 暂停 / 恢复 / 手动触发
- 执行日志查看

当前 `Timer` / `CreateTimerRequest` / `UpdateTimerRequest` 只包含任务自身调度信息与回调信息，不包含提醒配置。

#### 2. Electron 通知链路

当前 Electron 侧已有：

- `electron/notification.ts`
  - `showNotification`
  - `showDingIslandNotification`
  - `showSystemNotification`
  - `connectNotify`
- `electron/windows/islandWindow.ts`
  - 显示独立灵动岛窗口
- `src/components/DingIsland/DingIslandCard.vue`
  - 队列化展示灵动岛通知
- `electron/preload.ts` / `src/types/electron.d.ts`
  - 提供 `onIslandShow`、`sendNotificationAction`、`getConfig` 等桥接能力

#### 3. 当前不足

当前通知模型只有：

- `id`
- `title`
- `body`
- `level`
- `timestamp`

缺少：

- 通知来源类型
- 定时任务 `uuid`
- 任务名称 / 执行状态 / 执行时间
- 跳转目标
- 去重键 / 降噪信息
- 提醒渠道显式声明

这会导致灵动岛无法做“面向定时任务”的稳定交互。

## 需求范围

## 功能目标

### 1. 定时任务支持提醒配置

在“新增任务 / 编辑任务”中增加“提醒配置”区块，支持配置：

- 是否开启提醒
- 是否通过灵动岛提示
- 是否同时使用系统通知
- 提醒级别
- 在什么执行结果下提醒
- 高频任务的提醒冷却时间

### 2. 灵动岛可识别定时任务事件

当后端产生“定时任务提醒事件”后，Electron 能识别其为 `timer` 类通知，并展示：

- 任务名称
- 本次触发结果
- 触发时间 / 下次执行时间（如有）
- 快捷动作：查看、稍后提醒、关闭

### 3. 通知点击后可回到对应任务详情

灵动岛点击“查看”或卡片主体时：

- 唤起主窗口
- 跳转到定时任务页
- 自动定位到对应任务详情

## 非目标

本次不在 `web-ui` 内实现：

- 定时任务触发规则本身
- 通知是否生成的业务判定逻辑
- 服务端通知分发引擎
- 多端同步通知状态

这些能力应继续由后端负责，`web-ui` 只消费配置和展示结果。

## 核心设计

### 一、配置分层设计

为避免“所有逻辑都堆到 Electron 本地配置”或“所有逻辑都做成全局开关”，本方案采用两层配置：

#### 1. 任务级配置

任务级配置跟随单个 timer 持久化，由服务端存储，作为该任务是否允许提醒、如何提醒的主配置。

建议新增结构：

```ts
interface TimerNotifyConfig {
  enabled: boolean
  island_enabled: boolean
  system_enabled: boolean
  notify_on: 'all' | 'failure' | 'success' | 'manual'
  level: 'normal' | 'important' | 'urgent'
  cooldown_seconds: number
}
```

字段说明：

- `enabled`
  - 总开关，关闭后该任务不产生桌面提醒。
- `island_enabled`
  - 是否允许走灵动岛展示。
- `system_enabled`
  - 是否允许走系统通知。
- `notify_on`
  - 何种执行结果才提醒。
- `level`
  - 决定灵动岛展示强度和系统通知静音策略。
- `cooldown_seconds`
  - 高频 `periodic` 任务的降噪参数，防止刷屏。

#### 2. 客户端全局配置

客户端全局配置属于 Electron 本地偏好，不改定时任务语义，只做最终展示兜底：

```ts
interface AppConfig {
  notificationLevel: 'all' | 'important' | 'urgent'
  soundEnabled: boolean
  autoStart: boolean
  theme: 'light' | 'dark' | 'system'
  timerIslandEnabled: boolean
  timerSystemNotificationEnabled: boolean
}
```

新增建议：

- `timerIslandEnabled`
  - 全局是否允许展示“定时任务类灵动岛”
- `timerSystemNotificationEnabled`
  - 全局是否允许展示“定时任务类系统通知”

优先级建议：

`全局开关` > `任务级开关` > `事件级渠道`

即：

- 全局关闭时，无论单任务如何配置，都不展示。
- 全局开启时，再看任务是否开启对应渠道。
- 最终以事件 payload 中的推荐渠道为准进行展示。

### 二、通知事件模型设计

为让灵动岛真正理解“定时任务通知”，需要扩展当前 `NotificationData`。

建议统一为：

```ts
interface NotificationData {
  id: string
  source_type: 'timer' | 'system'
  biz_type: 'timer_triggered' | 'timer_success' | 'timer_failure'
  title: string
  body: string
  level: 'normal' | 'important' | 'urgent'
  timestamp: number
  channels?: {
    island?: boolean
    system?: boolean
  }
  timer?: {
    uuid: string
    name: string
    trigger_type: 'cron' | 'periodic' | 'delayed'
    run_status?: 'triggered' | 'success' | 'failure'
    run_at?: string
    next_run_at?: string
  }
  dedupe_key?: string
}
```

设计意图：

- `source_type`
  - 用于 Electron/渲染层区分“定时任务通知”和其他通知。
- `biz_type`
  - 用于决定标题、副文案、图标点样式和动作语义。
- `channels`
  - 服务端明确声明建议展示渠道，避免前端猜测。
- `timer`
  - 给灵动岛和回跳逻辑提供足够上下文。
- `dedupe_key`
  - 用于高频任务通知去重与短时间合并。

### 三、灵动岛展示策略

当前 `DingIslandCard.vue` 只负责基础队列展示，没有来源分类和动作语义。建议升级为“通用容器 + timer 专用展示规则”。

#### 1. 展示规则矩阵

| 级别 | 灵动岛行为 | 系统通知行为 | 建议场景 |
|------|------------|--------------|----------|
| `normal` | 自动消失，停留 4-5 秒 | 可静默 | 成功通知、低优先级提醒 |
| `important` | 自动消失，停留 8 秒 | 可同时展示 | 普通失败、需要用户关注 |
| `urgent` | 不自动消失或延长到 15 秒 | 强提醒 | 连续失败、一次性任务关键提醒 |

#### 2. 卡片内容建议

针对 `timer` 类通知：

- 主标题：任务名
- 副标题：执行结果 + 时间
- 辅助信息：下次执行时间（有则展示）
- 角标：失败 / 成功 / 手动触发

示例：

- `日报提醒` / `执行失败，23:30` / `下次执行：明天 09:00`
- `知识库同步` / `手动触发成功` / `队列中还有 2 条`

#### 3. 动作语义调整

当前动作为：

- `complete`
- `snooze`
- `view`

对于定时任务通知，建议在语义上解释为：

- `view`
  - 打开主窗口并跳转任务详情
- `snooze`
  - 仅对当前通知再次提醒，不修改 timer 本身
- `complete`
  - 作为“忽略/关闭本次提醒”处理

这样能兼容现有 IPC 结构，同时不把“完成任务”这种业务语义强加给 timer。

### 四、定时任务页面交互设计

#### 1. 表单入口

在 `TimerView.vue` 的“新增 / 编辑”弹窗中新增 `提醒配置` 区块。

建议字段顺序：

1. 开启提醒
2. 提醒渠道
   - 灵动岛
   - 系统通知
3. 提醒时机
   - 所有执行结果
   - 仅失败
   - 仅成功
   - 仅手动触发
4. 提醒级别
5. 冷却时间（秒）

#### 2. 详情页摘要

在右侧详情面板新增“提醒策略”摘要卡片，展示：

- 当前是否开启灵动岛提醒
- 当前提醒级别
- 提醒时机
- 冷却时间

这样用户不用重新打开编辑弹窗就能看到当前配置。

#### 3. 列表态轻提示

建议在任务列表中追加一个轻量提示标识：

- 已开启灵动岛提醒：显示 `岛` 或通知图标徽记
- 已开启系统通知：显示 `系统` 或铃铛徽记

这样用户在列表态即可判断哪些任务会推送提醒。

### 五、主窗口回跳与路由定位设计

灵动岛的核心体验不是“弹一下”，而是“能回到上下文”。

建议增加以下能力：

#### 1. 路由约定

使用定时任务页查询参数承接定位信息：

```text
#/timer?timerUuid=<uuid>
```

#### 2. 页面行为

`TimerView.vue` 在 `onMounted` 或路由监听中：

- 读取 `timerUuid`
- 自动选中对应 timer
- 自动加载详情和日志
- 如果列表未加载完成，先拉列表再选中

#### 3. Electron 侧动作处理

当收到 `view` 动作时：

- 唤起主窗口
- 导航到 `#/timer?timerUuid=<uuid>`
- 如果主窗口已在该页面，仅触发路由刷新或发送 IPC 定位事件

## 前后端契约建议

虽然本次只输出 `web-ui` 技术方案，但为了保证前端可落地，需要明确依赖的接口契约。

### 1. Timer 实体扩展

建议 `listTimers` / `getTimer` 返回中增加：

```ts
notify_config?: TimerNotifyConfig
```

### 2. 创建 / 更新接口扩展

建议 `createTimer` / `updateTimer` 请求体支持：

```ts
notify_config?: TimerNotifyConfig
```

### 3. 通知事件扩展

通知订阅接口返回的每条事件应尽量满足：

- 标准 `NotificationData`
- `source_type = 'timer'`
- 包含 `timer.uuid`
- 包含 `channels`
- 包含 `level`

如果后端已经具备提醒能力，但字段名与本方案不同，`web-ui` 端建议通过一个 `normalizeNotificationPayload()` 进行一次兼容映射。

## 交互流程

### 场景一：用户创建一个带灵动岛提醒的任务

1. 用户进入定时任务页。
2. 点击“新增任务”。
3. 在表单中填写调度信息和“提醒配置”。
4. `web-ui` 将 `notify_config` 一并提交给 `createTimer`。
5. 服务端保存该任务及其提醒策略。
6. 任务详情页展示提醒摘要。

### 场景二：任务执行失败，触发灵动岛提醒

1. 服务端根据 timer 的 `notify_config` 产生提醒事件。
2. Electron `connectNotify()` 收到通知事件。
3. `notification.ts` 进行标准化、过滤、去重。
4. 若允许展示灵动岛，则调用 `showIslandWindow(data)`。
5. `DingIslandCard.vue` 展示该 timer 卡片。
6. 用户点击“查看”后，主窗口跳转到 `#/timer?timerUuid=xxx`。
7. `TimerView.vue` 自动定位并加载该任务详情与日志。

### 场景三：高频 periodic 任务降噪

1. 高频任务连续触发多个结果事件。
2. 后端或 Electron 基于 `cooldown_seconds` / `dedupe_key` 进行限流。
3. 灵动岛只展示最近一次有效提醒。
4. 队列区显示 `+N`，避免刷屏。

## 风险与注意事项

### 1. 不要在 web-ui 内承接业务判定

`web-ui` 只能：

- 展示配置
- 提交配置
- 展示通知
- 处理客户端交互

不要在前端重做“哪些 timer 应该提醒”的业务规则，否则会和 Go 侧逻辑重复。

### 2. 高频 periodic 任务存在刷屏风险

如果没有 `cooldown_seconds` / `dedupe_key`，灵动岛会被高频任务淹没。

因此降噪应作为本方案的必要项，而不是可选项。

### 3. “完成”动作的语义不适合 timer

现有 `complete` 在 timer 场景下更适合作为“关闭本次提醒”。

后续若要更清晰，可以把动作语义逐步升级为：

- `dismiss`
- `snooze`
- `view_timer`

但为减少改动，第一阶段可以先兼容现有动作结构。

### 4. 路由回跳依赖主窗口导航能力

若当前 Electron 主窗口尚未提供可靠的“从通知跳转指定 hash 路由”的能力，需要在主进程中补一层导航桥接。

## 验收标准

- [ ] 新增/编辑定时任务时可配置提醒策略
- [ ] 定时任务详情可看到提醒配置摘要
- [ ] Electron 收到 timer 类事件后，可按配置决定是否展示灵动岛
- [ ] 灵动岛卡片能展示任务名称、执行结果、时间信息
- [ ] 点击灵动岛后可回到对应任务详情
- [ ] 高频 periodic 任务不会短时间连续刷出大量灵动岛卡片
- [ ] 全局关闭“定时任务灵动岛提醒”后，不再展示相关卡片

## 总结

本方案的关键不是“给灵动岛多弹一个提示”，而是把“定时任务提醒”正式建模成一条完整前端链路：

- 配置入口在 `TimerView`
- 配置主数据跟随 timer 持久化
- Electron 根据标准化事件决定展示策略
- 灵动岛展示 timer 上下文
- 点击后回到具体任务详情

这样才能让“灵动岛提示”从一个通用通知能力，升级为定时任务场景下真正可用的交互入口。
