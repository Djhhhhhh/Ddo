## 背景与目标

### 需求背景

目前CLI的RAG逻辑比较混乱，用户输入后缺乏清晰的决策链路。需要重构为标准化的意图-检索-生成流程：

```
用户输入
    ↓
① 意图识别（是否需要知识）
    ↓
② 路由决策（RAG / 直接回答 / 工具调用）
    ↓
③ （可选）RAG 检索
    ↓
④ 上下文构建（Prompt Assembly）
    ↓
⑤ LLM 生成回答
```

### 核心目标

1. **明确调用顺序**：CLI → Go → LLM，禁止直接 CLI → LLM
2. **分层职责清晰**：
   - CLI：交互层，负责用户反馈和状态展示
   - Go：编排层，负责意图识别、路由决策、RAG检索
   - LLM：推理层，负责意图分类、文本生成
3. **交互体验优化**：每一步都有清晰的反馈，用户知道当前处于哪个处理阶段

## 当前问题分析

### 现状流程

```
用户输入
    ↓
CLI (intent-router.ts)
    ↓
调用 NLP /api/v1/chat/nlp（Go代理到LLM）
    ↓
根据intent路由
    ├─ "chat" → 直接chat API
    ├─ "kb.xxx" → 切换模式或命令
    ├─ "timer.xxx" → 切换模式或命令
    └─ 其他 → 默认chat
```

### 现存问题

| 问题 | 影响 | 说明 |
|------|------|------|
| RAG与普通对话没有区分 | 用户体验差 | 不知道该不该走知识库检索 |
| 意图识别和RAG检索分离 | 调用冗余 | 两个独立API调用 |
| 没有"是否需要知识"的判断 | 检索盲目 | 问候语也会去检索 |
| CLI层逻辑过重 | 维护困难 | 路由决策分散在多个文件 |
| 流式体验不连贯 | 视觉割裂 | 思考中、检索中、生成中状态混乱 |

### 核心组件介绍

#### 1. 意图识别服务 (Intent Recognition)

**职责**：判断用户输入的意图类型和知识需求

**输入**：`{ text: string, conversation_id?: string }`

**输出**：
```go
type IntentResult struct {
    Intent        string   `json:"intent"`        // 主意图: chat, kb_add, kb_search, timer_add...
    SubIntent     string   `json:"sub_intent"`    // 子意图: create_hourly, search_recent...
    NeedKnowledge bool     `json:"need_knowledge"`// 是否需要知识库检索
    Confidence    float64  `json:"confidence"`    // 置信度
    Entities      []Entity `json:"entities"`      // 提取的实体
    Parameters    map[string]interface{} `json:"parameters"`
    SuggestedReply string  `json:"suggested_reply"`// 给CLI的即时回复
}
```

#### 2. 路由决策引擎 (Route Decision)

**决策矩阵**：

| NeedKnowledge | Intent | Decision | Action |
|--------------|--------|----------|--------|
| true | 任意 | RAG检索 | 向量搜索 + LLM生成 |
| false | chat | 直接回答 | LLM闲聊 |
| false | kb_add | 工具调用 | 打开KB添加交互 |
| false | timer_add | 工具调用 | 打开Timer添加交互 |
| low_confidence | 任意 | 澄清询问 | 向用户确认意图 |

#### 3. RAG Pipeline

**流程**：
```
用户Query
    ↓
Embedding（复用现有embedder）
    ↓
Vector Search（top_k=5, min_score=0.7）
    ↓
结果重排序（可选）
    ↓
Prompt组装（Context + Question）
    ↓
LLM流式生成
```

### CLI代码结构

```typescript
// src/repl/conversation-handler.ts

interface ConversationState {
  status: 'idle' | 'analyzing' | 'retrieving' | 'generating';
  intent?: IntentResult;
  sources?: Source[];
}

class ConversationHandler {
  private state: ConversationState = { status: 'idle' };

  async handleUserInput(input: string): Promise<void> {
    // 1. 显示分析中
    this.showAnalyzing();
    
    // 2. 调用 Go API 进行意图识别
    const intent = await this.api.detectIntent(input);
    
    // 3. 路由决策
    if (intent.needClarification) {
      this.askClarification(intent);
      return;
    }
    
    if (intent.needKnowledge) {
      // 4. 执行 RAG 流程
      await this.handleRAGQuery(input, intent);
    } else if (intent.intent.startsWith('tool.')) {
      // 5. 执行工具调用
      await this.handleToolCall(intent);
    } else {
      // 6. 直接对话
      await this.handleDirectChat(input);
    }
  }

  private async handleRAGQuery(query: string, intent: IntentResult): Promise<void> {
    this.showRetrieving();
    
    // 调用统一的 chat-with-rag API
    const stream = await this.api.chatWithRAG({
      query,
      conversationId: this.conversationId,
      stream: true,
    });
    
    this.showGenerating();
    await this.streamResponse(stream);
  }
}
```

### 核心服务实现

```go
// internal/application/service/conversation_service.go

package service

type ConversationService interface {
    // 处理用户查询（编排入口）
    ProcessQuery(ctx context.Context, req *QueryRequest) (*QueryResponse, error)
    
    // 流式处理
    ProcessQueryStream(ctx context.Context, req *QueryRequest) (<-chan StreamEvent, error)
}

type conversationService struct {
    intentSvc   IntentRecognitionService   // 意图识别
    ragSvc      RAGService                 // RAG服务
    chatSvc     ChatService                // 直接对话
    toolRouter  ToolRouter                 // 工具路由
}

func (s *conversationService) ProcessQuery(ctx context.Context, req *QueryRequest) (*QueryResponse, error) {
    // Step 1: 意图识别
    intent, err := s.intentSvc.Recognize(ctx, req.Query)
    if err != nil {
        return nil, fmt.Errorf("intent recognition failed: %w", err)
    }
    
    // Step 2: 路由决策
    decision := s.makeDecision(intent, req.KBPriority)
    
    switch decision {
    case DecisionRAG:
        return s.handleRAG(ctx, req, intent)
    case DecisionChat:
        return s.handleChat(ctx, req)
    case DecisionTool:
        return s.handleTool(ctx, req, intent)
    default:
        return nil, fmt.Errorf("unknown decision: %v", decision)
    }
}

func (s *conversationService) handleRAG(ctx context.Context, req *QueryRequest, intent *Intent) (*QueryResponse, error) {
    // Step 3: RAG检索（调用llm-py的RAG API）
    ragReq := &RAGRequest{
        Question: req.Query,
        ContextLimit: 5,
        MinScore: 0.7,
    }
    
    ragResult, err := s.ragSvc.Ask(ctx, ragReq)
    if err != nil {
        // RAG失败，降级到直接对话
        return s.handleChat(ctx, req)
    }
    
    return &QueryResponse{
        Decision:      DecisionRAG,
        Intent:        intent,
        Answer:        ragResult.Answer,
        Sources:       ragResult.Sources,
        RetrievedDocs: ragResult.Documents,
    }, nil
}
```

### C. 决策矩阵参考

```go
type Decision int

const (
    DecisionRAG Decision = iota    // 需要RAG检索
    DecisionChat                    // 直接聊天
    DecisionTool                    // 工具调用
    DecisionClarify                 // 需要澄清
)

func makeDecision(intent *Intent, kbPriority bool) Decision {
    // 知识优先模式
    if kbPriority && intent.Confidence > 0.5 {
        return DecisionRAG
    }
    
    // 高置信度意图
    if intent.Confidence > 0.8 {
        switch intent.Intent {
        case IntentKnowledgeQuery:
            return DecisionRAG
        case IntentKnowledgeAdd, IntentTimerAdd, IntentMCPAdd:
            return DecisionTool
        case IntentChat:
            return DecisionChat
        }
    }
    
    // 低置信度
    if intent.Confidence < 0.5 {
        return DecisionClarify
    }
    
    // 根据need_knowledge
    if intent.NeedKnowledge {
        return DecisionRAG
    }
    
    return DecisionChat
}
```
