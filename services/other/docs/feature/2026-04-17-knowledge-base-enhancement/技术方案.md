# 知识库增强功能技术方案

## 1. 背景与目标

### 1.1 背景
现有系统包含三个服务：CLI、server-go、llm-py。需要在 `/kb-add` 命令中增加知识添加编辑，支持来源追溯、智能分类和标签。

### 1.2 目标
- CLI 自动装配 `source` 字段
- AI 自动分析生成 `tags` 和 `category`
- 支持多对多 category 关系
- 三个服务修改后保证一致性

---

## 2. 整体架构

### 2.1 服务调用时序图

```
┌─────────┐    POST /kb-add     ┌───────────┐    创建知识      ┌───────────┐    AI分析      ┌─────────┐
│   CLI   │ ──────────────────▶ │ server-go  │ ────────────────▶│   MySQL   │ ◀──────────────│  llm-py │
│         │                     │            │                 │           │                │         │
│ source: │                     │ source: ✓  │                 │           │                │         │
│ content │                     │ category: ?│                 │           │                │         │
│         │                     │ tags: ?    │                 │           │                │         │
└─────────┘                     └───────────┘                 └───────────┘                └─────────┘
                                       ▲
                                       │ GET /nlp/analyze
                                       │ (CLI 不直接调用)
```


**调用链路（严格）**：`CLI → server-go → llm-py`

### 2.2 各服务职责

| 服务 | 职责 |
|------|------|
| **cli** | 修改 `/kb-add` 命令，自动添加 `source` 字段，调用 server-go API |
| **server-go** | 新增 category 表及多对多关联，修改知识创建接口，调用 llm-py 分析 tags/category |
| **llm-py** | 新增 NLP 接口，输入知识内容，输出推荐的 tags 列表和 category 列表 |

## 3. 数据库设计

### 3.1 新增表

#### 3.1.1 `categories` 表

```sql
CREATE TABLE categories (
    id          VARCHAR(36) PRIMARY KEY,
    name        VARCHAR(100) NOT NULL UNIQUE,
    description TEXT,
    created_at  DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at  DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_name (name)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

#### 3.1.2 `knowledge_categories` 多对多关联表

```sql
CREATE TABLE knowledge_categories (
    knowledge_id VARCHAR(36) NOT NULL,
    category_id  VARCHAR(36) NOT NULL,
    created_at   DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (knowledge_id, category_id),
    INDEX idx_category (category_id),
    FOREIGN KEY (knowledge_id) REFERENCES knowledge(uuid) ON DELETE CASCADE,
    FOREIGN KEY (category_id) REFERENCES categories(id) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

### 3.2 现有表变更

#### 3.2.1 `knowledge` 表字段说明

| 字段 | 类型 | 说明 |
|------|------|------|
| `uuid` | VARCHAR(36) PK | 唯一标识 |
| `title` | VARCHAR(255) | 标题 |
| `content` | TEXT | 正文内容 |
| `category` | VARCHAR(100) | **主分类**（冗余字段，用于快速查询） |
| `tags` | JSON | 标签数组 |
| `source` | VARCHAR(500) | 来源，如 `"cli"` |
| `embedding_id` | VARCHAR(36) | llm-py 中的向量 ID |
| `status` | VARCHAR(20) | 状态 |
| `created_at` | DATETIME | 创建时间 |
| `updated_at` | DATETIME | 更新时间 |
| `deleted_at` | DATETIME | 软删除时间 |

> **注意**：`category` 保留作为**主分类**（第一个分类），真实关系存储在 `knowledge_categories` 表中。

## 4. API 设计

### 4.1 llm-py：新增 `/nlp/analyze` 接口

**请求**：

```json
POST /api/nlp/analyze
{
  "content": "Python 异步编程核心概念：async/await、协程、事件循环",
  "title": "Python 异步编程入门",
  "context": "knowledge_base"
}
```


**响应**：

```json
{
  "tags": ["python", "async", "coroutine", "event-loop"],
  "categories": ["编程语言", "异步编程"],
  "is_new_categories": [false, true],
  "suggested_reply": "已将内容添加到【编程语言】分类，并打上了相关标签"
}
```

### 4.3 server-go：新增 category 管理接口

```

```

### 4.4 CLI：修改 `/kb-add` 命令

```

```

- **自动添加 `source: "cli"`**
- 调用 server-go `/api/v1/knowledge` 接口
- 不直接调用 llm-py

## 5. 各服务详细修改点

### 5.1 services/llm-py

#### 5.1.1 新增文件

**`app/api/analyze.py`**

```python
"""
Knowledge Analysis API
分析知识内容，提取 tags 和 category。
"""

from fastapi import APIRouter, HTTPException, status
from pydantic import BaseModel, Field
from typing import List, Optional

from app.core.llm_factory import get_llm_factory, LLMFactoryError
from app.utils.logger import get_logger


router = APIRouter()


class AnalyzeRequest(BaseModel):
    """知识分析请求。"""
    content: str = Field(..., description="知识内容")
    title: Optional[str] = Field(None, description="知识标题")
    context: Optional[str] = Field("knowledge_base", description="分析上下文")


class AnalyzeResponse(BaseModel):
    """知识分析响应。"""
    tags: List[str] = Field(..., description="推荐标签列表")
    categories: List[str] = Field(..., description="推荐分类列表")
    is_new_categories: List[bool] = Field(..., description="对应分类是否为新创建")
    suggested_reply: str = Field(default="", description="建议回复")


@router.post("/analyze", response_model=AnalyzeResponse, status_code=status.HTTP_200_OK)
async def analyze_knowledge(request: AnalyzeRequest) -> AnalyzeResponse:
    """
    分析知识内容，提取标签和分类。

    Args:
        request: 包含 content 和 title 的分析请求。

    Returns:
        分析结果，包含 tags、categories、是否为新分类、建议回复。
    """
    factory = get_llm_factory()

    try:
        # 构建 prompt
        text_to_analyze = f"标题: {request.title}\n内容: {request.content}" if request.title else request.content

        # 使用 LLM 分析（通过 create_nlp_intent_chain 扩展或新建 chain）
        chain = factory.create_knowledge_analysis_chain(context=request.context)

        result = await chain.ainvoke({"text": text_to_analyze})

        return AnalyzeResponse(
            tags=result.get("tags", []),
            categories=result.get("categories", []),
            is_new_categories=result.get("is_new_categories", []),
            suggested_reply=result.get("suggested_reply", ""),
        )

    except LLMFactoryError as e:
        raise HTTPException(status_code=status.HTTP_400_BAD_REQUEST, detail=str(e))
    except Exception as e:
        raise HTTPException(status_code=status.HTTP_502_BAD_GATEWAY, detail=str(e))
```

## 6. AI 分析逻辑说明（llm-py 的提示词设计）

### 6.1 Prompt 设计

```
你是一个知识库管理员。请分析以下知识内容，提取：

1. tags（标签）：3~8个，简短描述性词汇，用于"精确检索"
2. categories（分类）：1~3个，宽泛组织词汇，用于"按领域浏览"


输出 JSON：
{
  "tags": ["tag1", "tag2", ...],
  "categories": ["分类1", "分类2", ...],
  "is_new_categories": [false, true, ...],  // 是否需要新建该分类
  "suggested_reply": "..."
}

知识内容：
{text}
```


### 6.2 优先级规则

分析时，先判断 categories，再提取 tags。

## 7. 错误处理与边界情况

### 7.1 AI 不可用
- llm-py 返回 502 或超时
- server-go 降级：**不阻止**知识创建，tags 置空，category 保留传入值
- 记录错误日志，后续可手动补全

### 7.2 Category 创建冲突
- 并发创建同名 category 导致唯一索引冲突
- server-go 处理：`ON DUPLICATE KEY` 忽略或回查已有记录

### 7.3 Category 数据不一致
- 主分类字段（冗余）与 `knowledge_categories` 表不一致
- 规则：以 `knowledge_categories` 为准，`category` 字段仅用于显示

### 7.4 AI 无法判断分类
- 若 AI 无法判断，归类为 **"待分类"**
- 自动创建该分类（is_new_categories=true）
