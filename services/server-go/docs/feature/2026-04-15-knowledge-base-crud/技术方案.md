# 知识库 CRUD 与 RAG 代理 - 技术方案

## 背景与目标

server-go 作为 Ddo 平台的核心网关，需要实现知识库的元数据管理功能。根据架构设计，server-go 负责知识库的 CRUD 操作（存储在 MySQL），而 RAG 检索和向量生成交由 llm-py 处理。

**为什么拆分？**
- server-go 专注于元数据管理、权限控制和 API 聚合
- llm-py 专注于 AI 能力，包括向量生成和语义检索
- 避免重复实现向量数据库逻辑

## 需求概述

实现完整的知识库管理功能：

1. **知识条目 CRUD**：元数据的增删改查（存储在 MySQL）
2. **RAG 代理**：将检索和问答请求转发到 llm-py
3. **分类和标签管理**：支持按分类和标签筛选
4. **向量管理**：与 llm-py 配合完成文档嵌入

## 技术方案

### 方案选型

| 层级 | 技术选型 | 说明 |
|------|----------|------|
| HTTP 框架 | Gin | 路由分组、中间件、统一响应 |
| 数据库 | MySQL + GORM | 元数据存储，已完成的 p2-12 |
| RAG 代理 | HTTP Client | 转发到 llm-py `/api/rag/*` |
| 配置 | 环境变量 | `DDO_LLM_HOST` 配置 llm-py 地址 |

### 核心实现步骤

1. **知识库 Repository 层** (`internal/db/repository/knowledge_repo.go`)
   - 封装 GORM 查询：分页、条件筛选、关联查询
   - 支持按 UUID、分类、标签查询
   - 软删除支持

2. **知识库 UseCase 层** (`internal/application/usecase/knowledge/...`)
   - CreateKnowledgeUseCase: 创建知识条目，同时调用 llm-py 生成向量
   - ListKnowledgeUseCase: 分页查询知识列表
   - GetKnowledgeUseCase: 获取单条知识详情
   - DeleteKnowledgeUseCase: 软删除知识条目
   - SearchKnowledgeUseCase: 代理到 llm-py 进行语义搜索
   - AskKnowledgeUseCase: 代理到 llm-py 进行 RAG 问答

3. **RAG 代理服务** (`internal/application/service/rag_proxy.go`)
   - 封装 llm-py HTTP 调用
   - 接口：EmbedDocument, SearchVector, AskRAG
   - 错误处理和重试机制
   - 超时控制（默认 30s）

4. **HTTP Handler 层** (`internal/interfaces/http/handler/knowledge_handler.go`)
   - 实现 RESTful API 端点
   - 请求参数验证
   - 统一响应格式转换

5. **路由注册** (`internal/interfaces/http/router.go`)
   - 注册知识库相关路由
   - 添加知识库模块到 Wire 依赖注入

## 接口定义

### 知识条目管理

| 接口 | 方法 | 路径 | 功能 |
|------|------|------|------|
| 创建知识 | POST | `/api/v1/knowledge` | 创建知识条目，自动调用 llm-py 嵌入 |
| 查询列表 | GET | `/api/v1/knowledge` | 分页查询，支持分类、标签筛选 |
| 获取单条 | GET | `/api/v1/knowledge/:uuid` | 获取知识详情 |
| 删除知识 | POST | `/api/v1/knowledge/:uuid/delete` | 软删除知识条目 |

### RAG 代理接口

| 接口 | 方法 | 路径 | 功能 |
|------|------|------|------|
| 语义搜索 | GET | `/api/v1/knowledge/search` | 代理到 llm-py 语义搜索 |
| RAG 问答 | POST | `/api/v1/knowledge/ask` | 代理到 llm-py 生成回答 |

### 请求/响应示例

**创建知识 - 请求**
```json
POST /api/v1/knowledge
{
  "title": "Golang最佳实践",
  "content": "内容正文...",
  "category": "编程语言",
  "tags": ["go", "backend"],
  "source": "https://example.com/doc"
}
```

**语义搜索 - 请求**
```
GET /api/v1/knowledge/search?q=Go语言并发模式&limit=5
```

**RAG 问答 - 请求**
```json
POST /api/v1/knowledge/ask
{
  "question": "请解释Go的Channels",
  "context_limit": 3
}
```

## RAG 代理调用流程

**创建知识时调用 llm-py：**
```
server-go                    llm-py
   |                           |
   |--> POST /api/rag/embed -->|
   |                           |
   |<-- embedding_id ----------|
   |                           |
   存储到 MySQL + embedding_id
```

**语义搜索时调用 llm-py：**
```
server-go                        llm-py
   |                               |
   |--> GET /api/rag/search?q=xxx-->|
   |                               |
   |<-- [doc1, doc2, ...] --------|
   |                               |
合并 MySQL 元数据返回
```

## LLM-PY 接口约定

**Embed 接口**
- 路径: `POST /api/rag/embed`
- 请求: `{ "content": "文本内容", "metadata": {...} }`
- 响应: `{ "embedding_id": "uuid", "dimensions": 1536 }`

**Search 接口**
- 路径: `GET /api/rag/search?q={query}&limit={n}`
- 响应: `[{ "embedding_id": "...", "score": 0.95, "content": "..." }]`

**Ask 接口**
- 路径: `POST /api/rag/ask`
- 请求: `{ "question": "...", "context_docs": [...] }`
- 响应: `{ "answer": "...", "sources": [...] }`
