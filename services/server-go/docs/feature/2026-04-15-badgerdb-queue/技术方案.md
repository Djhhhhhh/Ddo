# server-go BadgerDB 队列 - 技术方案

## 背景与目标

server-go 需要支持定时任务调度、延时任务执行、以及内部消息传递。考虑到 Ddo 是本地单机运行的应用，使用嵌入式数据库 BadgerDB（纯 Go 实现的高性能 LSM-tree 存储）作为消息队列，避免引入外部依赖（如 Redis/RabbitMQ）。

## 需求概述

1. **嵌入式消息队列**: 基于 BadgerDB 实现轻量级消息队列，无需外部服务
2. **延时任务支持**: 支持定时投递消息，用于定时任务触发
3. **Topic 订阅**: 支持按 topic 分类消息，消费者可按 topic 订阅
4. **优先级队列**: 支持消息优先级，重要消息优先处理
5. **持久化保证**: 消息落盘，服务重启不丢失

## 技术方案

### 方案选型

- **存储引擎**: BadgerDB v4（纯 Go 实现，性能优异，支持 KV 存储和迭代器）
- **数据序列化**: JSON（可读性好，易于调试）或 MessagePack（性能更优）
- **调度器**: 内部 goroutine 轮询，时间精度秒级
- **并发模型**: 单写多读，使用 channel 进行消息分发

### 核心实现步骤

1. **初始化队列模块结构**
   - 创建 `internal/queue/queue.go` 定义 Queue 接口
   - 创建 `internal/queue/badger_queue.go` 实现 BadgerDB 版本

2. **BadgerDB 实例创建**
   - 数据目录: `~/.ddo/data/badger/queue`
   - 配置选项: `SyncWrites=true`（保证持久化），`Logger=null`（生产环境静默）
   - 启动时打开，关闭时调用 `db.Close()`

3. **消息结构设计**
   ```go
   type Message struct {
       ID        string          `json:"id"`        // UUID
       Topic     string          `json:"topic"`     // 主题，如 "timer", "notify"
       Payload   json.RawMessage `json:"payload"`   // 消息体
       Priority  int             `json:"priority"`  // 优先级 0-9，数字越小越优先
       DelayAt   *time.Time      `json:"delay_at"`  // 延时执行时间，nil 表示立即执行
       CreatedAt time.Time       `json:"created_at"` // 创建时间
       Attempts  int             `json:"attempts"`  // 重试次数
   }
   ```

4. **核心操作实现**
   - `Publish(msg *Message) error` - 发布消息（立即或延时）
   - `Subscribe(topics []string, handler MessageHandler) error` - 订阅 topic
   - `Ack(msgID string) error` - 确认消息已处理（删除）
   - `Nack(msgID string, delay time.Duration) error` - 否定确认，重新投递

5. **延时任务调度器**
   - 启动后台 goroutine 轮询延时队列
   - 轮询间隔: 1 秒
   - Key 设计: `delayed:{timestamp}:{uuid}`，按时间排序便于扫描
   - 到达执行时间的消息移动到 `pending:{topic}:{priority}:{uuid}`

6. **消费者实现**
   - 每个 Subscriber 启动独立 goroutine
   - 从 `pending:{topic}:*` 前缀扫描消息
   - 按优先级排序，高优先级优先处理
   - 并发数控制：每个 topic 限制最大并发处理数

7. **Wire 依赖注入集成**
   - 在 `wire.go` 中添加 `queue.NewBadgerQueue` Provider
   - Queue 生命周期与 Server 绑定，关闭时调用 `queue.Close()`

8. **定时任务集成（预留接口）**
   - 在 `internal/scheduler/` 中预留调度器
   - 定时任务触发时调用 `queue.Publish(&Message{Topic: "timer", ...})`
   - 消费者从 "timer" topic 接收消息并执行 HTTP 回调

## 接口定义

```go
// Queue 消息队列接口
type Queue interface {
    // Publish 发布消息，delay 为 0 表示立即投递
    Publish(ctx context.Context, topic string, payload []byte, priority int, delay time.Duration) error
    
    // Subscribe 订阅指定 topics，handler 处理消息
    Subscribe(topics []string, handler MessageHandler) error
    
    // Close 关闭队列连接
    Close() error
}

// MessageHandler 消息处理函数
type MessageHandler func(ctx context.Context, msg *Message) error

// Message 消息结构
type Message struct {
    ID        string
    Topic     string
    Payload   []byte
    Priority  int
    DelayAt   *time.Time
    CreatedAt time.Time
    Attempts  int
}
```

## BadgerDB Key 设计

```
# 延时消息（等待执行）
delayed:{unix_timestamp}:{uuid} -> Message (JSON)

# 待处理消息（可消费）
pending:{topic}:{priority}:{uuid} -> Message (JSON)

# 处理中消息（防止丢失）
processing:{consumer_id}:{uuid} -> Message (JSON) + 超时时间

# 元数据
meta:topics -> ["timer", "notify", ...]  # 已注册 topic 列表
meta:stats:{topic} -> {published, consumed, failed}  # 统计信息
```

## 性能优化

1. **批量写入**: 使用 BadgerDB 的 `WriteBatch` 批量写入消息
2. **前缀扫描**: 利用 LSM-tree 的有序性，前缀扫描延时队列
3. **TTL 清理**: 使用 BadgerDB 的 TTL 功能自动清理过期消息（如 30 天前的已消费消息）
4. **内存映射**: 依赖 BadgerDB 的 memory-mapped file 提高读取性能

## 错误处理

1. **磁盘满**: 返回 `ErrQueueFull`，触发告警（预留接口）
2. **消息处理失败**: 超过最大重试次数（默认 3 次）后进入死信队列 `dlq:{topic}`
3. **消费者 panic**: 捕获 panic 并记录日志，不影响其他消费者
4. **幂等性**: 消息 ID 去重，防止重复消费
