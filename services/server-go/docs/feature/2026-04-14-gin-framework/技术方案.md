# server-go: Gin 框架基础 - 技术方案（DDD 架构）

## 背景与目标

server-go 是 Ddo 平台的核心网关服务，负责提供统一的 API 路由、定时任务调度、消息队列、知识库管理和 MCP 管理能力。本任务为 server-go 服务的初始化阶段，目标是搭建基于 **DDD（领域驱动设计）** 架构的 Go 项目基础框架，使用 Gin 作为 HTTP 传输层，为后续业务功能开发奠定清晰的领域边界和分层结构。

## 需求概述

根据 MVP 文档和 DDD 架构设计，本阶段需要实现：

1. **初始化 DDD 项目结构**：按照领域分层创建清晰的项目目录
2. **配置 Gin HTTP 服务**：作为基础设施层的传输适配器
3. **实现领域层基础**：定义实体、值对象、领域服务的组织结构
4. **搭建应用层框架**：用例（UseCase）和应用服务的基座
5. **配置服务启动**：命令行参数和配置文件支持，实现优雅关闭

### DDD 分层架构设计

采用经典 DDD 四层架构，依赖关系向内指向领域层：

```
┌─────────────────────────────────────────────────────────────┐
│                      接口层 (Interfaces)                      │
│     HTTP Handler / DTO / 中间件 / 路由注册                     │
│                    ↓ 依赖应用层                                │
├─────────────────────────────────────────────────────────────┤
│                      应用层 (Application)                     │
│     UseCase / AppService / 事务控制 / 用例编排                 │
│                    ↓ 依赖领域层                                │
├─────────────────────────────────────────────────────────────┤
│                      领域层 (Domain) ★ 核心                   │
│     Entity / ValueObject / DomainService / Repository接口      │
│     DomainEvent / Aggregate / 业务规则                         │
│                    ↓ 不依赖外层                                │
├─────────────────────────────────────────────────────────────┤
│                      基础设施层 (Infrastructure)               │
│     MySQLRepo / BadgerQueue / GinServer / MCPClient           │
│     Config / Logger / 外部 API 客户端                          │
└─────────────────────────────────────────────────────────────┘
```

### 各层职责

#### 1. 领域层（Domain）- 核心业务
- **实体（Entity）**：具有唯一标识和生命周期的业务对象，如 `Timer`、`Knowledge`
- **值对象（ValueObject）**：无标识的不可变对象，如 `CronExpr`、`CallbackURL`
- **领域服务（DomainService）**：跨实体的业务逻辑，如 `TimerSchedulerService`
- **仓库接口（Repository）**：定义数据访问接口，由基础设施层实现
- **领域事件（DomainEvent）**：记录业务状态变更，如 `TimerTriggeredEvent`
- **聚合根（Aggregate）**：一致性边界，如 `KnowledgeAggregate`

#### 2. 应用层（Application）- 用例编排
- **用例（UseCase）**：一个完整的业务流程，如 `CreateTimerUseCase`
- **应用服务（AppService）**：协调领域对象完成用例，控制事务边界
- **DTO（Data Transfer Object）**：跨层数据传输对象
- **事务控制**：确保业务操作的原子性

#### 3. 接口层（Interfaces）- 协议适配
- **HTTP Handler**：Gin 的请求处理器，将 HTTP 请求转换为应用层用例
- **DTO**：请求/响应数据结构，负责数据校验和转换
- **中间件**：鉴权（预留）、日志、异常恢复、CORS
- **路由注册**：定义 URL 路由与 Handler 的映射

#### 4. 基础设施层（Infrastructure）- 技术实现
- **RepoImpl**：仓库接口的具体实现（MySQL、BadgerDB）
- **GinServer**：HTTP 服务器实现
- **Config**：配置文件解析（Viper）
- **Logger**：日志实现（Zap）
- **MQ**：消息队列实现（BadgerDB 延时队列）
- **MCPClient**：MCP 客户端实现

### 依赖关系原则

```
依赖方向：外层 → 内层

接口层 ────────────→ 应用层 ────────────→ 领域层
    ↓                    ↓                   ↑
    └────────────────────┴───────────────────┘
                    依赖接口
                    不依赖具体实现
```

## 核心实现步骤

### 步骤 1: 初始化 DDD 项目目录结构

1. 在项目根目录执行 `go mod init github.com/ddo/server-go`
2. 按照 DDD 四层架构创建项目目录
3. 初始化 Wire 依赖注入配置

### 步骤 2: 实现领域层基础（Domain Layer）

1. 创建领域共享包 `internal/domain/common/`：
   - `entity.go`：定义实体基类（含 ID、CreatedAt、UpdatedAt）
   - `event.go`：定义领域事件接口和事件总线
   - `errors.go`：定义领域错误（DomainError）

2. 创建第一个聚合根示例 `internal/domain/health/`：
   - `aggregate.go`：Health 聚合根（演示用）
   - `valueobject.go`：HealthStatus 值对象

### 步骤 3: 实现应用层框架（Application Layer）

1. 创建 `internal/application/dto/`：定义数据传输对象
2. 创建 `internal/application/service/`：定义应用服务接口
3. 创建 `internal/application/usecase/`：定义用例接口和实现
4. 创建 `internal/application/result/`：统一响应结果封装

### 步骤 4: 实现接口层（Interfaces Layer）

1. 创建 `internal/interfaces/http/handler/`：HTTP 请求处理器
2. 创建 `internal/interfaces/http/middleware/`：中间件（Recovery、Logger、CORS）
3. 创建 `internal/interfaces/http/dto/`：HTTP 请求/响应 DTO
4. 创建 `internal/interfaces/http/router.go`：路由注册

### 步骤 5: 实现基础设施层（Infrastructure Layer）

1. 创建 `internal/infrastructure/config/`：Viper 配置实现
2. 创建 `internal/infrastructure/logger/`：Zap 日志实现
3. 创建 `internal/infrastructure/server/`：Gin HTTP 服务器适配器
4. 创建 `wire.go`：Wire 依赖注入配置

### 步骤 6: 实现启动入口和优雅关闭

1. 创建 `cmd/server/main.go`：服务启动入口
2. 创建 `internal/bootstrap/app.go`：应用生命周期管理
3. 实现优雅关闭：监听 SIGINT/SIGTERM，确保请求处理完毕后退出

### 调用链路示例

```
HTTP Request
    ↓
[Interfaces] Handler (参数校验、DTO 转换)
    ↓
[Application] UseCase (事务控制)
    ↓
[Domain] DomainService → Entity → Repository(接口)
    ↓
[Infrastructure] RepositoryImpl (MySQL/Cache)
    ↓
[Application] UseCase (提交事务)
    ↓
[Interfaces] Handler (响应封装)
    ↓
HTTP Response
```

## DDD 核心代码示例

### 领域层 - 领域错误定义

```go
// internal/domain/common/errors.go
package common

type DomainError struct {
    Code    string
    Message string
}

func (e DomainError) Error() string {
    return e.Message
}

var (
    ErrEntityNotFound = DomainError{Code: "ENTITY_NOT_FOUND", Message: "实体不存在"}
    ErrInvalidState   = DomainError{Code: "INVALID_STATE", Message: "无效的状态变更"}
)
```

### 应用层 - 用例接口定义

```go
// internal/application/usecase/health/check_health.go
package health

type CheckHealthUseCase interface {
    Execute(ctx context.Context) (*CheckHealthResult, error)
}

type CheckHealthResult struct {
    Status    string
    Version   string
    Timestamp time.Time
}
```

### 接口层 - HTTP Handler

```go
// internal/interfaces/http/handler/health_handler.go
package handler

type HealthHandler struct {
    checkHealthUC usecase.CheckHealthUseCase
}

func NewHealthHandler(checkHealthUC usecase.CheckHealthUseCase) *HealthHandler {
    return &HealthHandler{checkHealthUC: checkHealthUC}
}

func (h *HealthHandler) HealthCheck(c *gin.Context) {
    result, err := h.checkHealthUC.Execute(c.Request.Context())
    if err != nil {
        c.JSON(500, response.Error(err))
        return
    }
    c.JSON(200, response.Success(result))
}
```
