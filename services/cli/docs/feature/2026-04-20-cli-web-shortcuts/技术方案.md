# CLI Web 快捷命令与可视化入口统一 - 技术方案

## 背景与目标

当前 CLI 已经具备 `/status`、`/timer`、`/mcp` 等 REPL 快捷命令，也具备 `timer-add`、`mcp-add` 这类交互式参数采集能力，但缺少“直接拉起对应 Web 页面”的统一入口。用户在执行状态查看、MCP 配置、定时任务管理时，CLI 与 Dashboard 之间仍需要手工切换，操作链路偏长。

本次目标是在 `services/cli` 内补齐一组 `/xx-web` 命令，使 REPL 能快速打开 Web Dashboard 的对应页面；同时把 CLI 中与 MCP、定时任务相关的交互流与 Web 表单保持一致，形成“CLI 可直达页面、CLI 交互字段与 Web 语义一致”的统一体验。

## 需求概述

本任务聚焦 CLI 单服务实现，包含以下能力：

1. 新增 `/xx-web` 系列命令，用于打开 Dashboard 对应页面。
2. 至少覆盖 `/status-web`，并为 `/timer-web`、`/mcp-web` 预留统一实现方式。
3. CLI 内的 MCP、定时任务操作流程需要与 Web 端页面信息架构对齐，避免 CLI 字段命名和页面表单语义不一致。
4. 命令应优先复用已有 REPL 命令注册体系、快捷命令模式和配置中的 `endpoints.webUi` 地址。
5. 方案必须兼容 Windows/macOS，避免写死平台相关命令。

## 技术方案

### 方案选型

- 技术选型: 基于现有 TypeScript CLI 结构，在 `src/repl/commands/` 中新增 Web 快捷命令文件，并增加一个浏览器打开工具模块；URL 基址优先读取配置文件中的 `endpoints.webUi`，回退到 `http://localhost:3000`。
- 设计模式: 延续 `mode-switch.ts` 的“快捷命令工厂”思路，抽象 `createWebShortcutCommand`，把命令名、页面路径、说明文案统一注册，减少重复代码。
- 依赖库/工具: 优先使用 Node 原生命令分平台拉起浏览器（Windows `start`、macOS `open`、Linux `xdg-open`）；若后续稳定性不足，再评估引入轻量库如 `open`。

### 核心实现步骤

1. 在 CLI 内新增 Web 页面打开能力。
   - 新增 `src/utils/open-url.ts` 一类工具模块，封装跨平台打开浏览器逻辑。
   - 读取 `~/.ddo/config.yaml` 中的 `endpoints.webUi`，并支持从环境变量或默认值回退。
   - 对 URL 进行规范化拼接，确保 `/status-web`、`/timer-web`、`/mcp-web` 能稳定落到 Dashboard 子路由。

2. 新增 `/xx-web` 命令并接入 REPL 注册表。
   - 新增 `src/repl/commands/web-shortcuts.ts`。
   - 提供 `createWebShortcutCommand(name, route, description)` 工厂函数，统一生成 `/status-web`、`/timer-web`、`/mcp-web` 等命令。
   - 在 `src/repl/index.ts` 中注册新命令，保持现有命令装配方式不变。

3. 统一 CLI 与 Web 的页面映射关系。
   - `/status-web` -> Dashboard 总览页。
   - `/timer-web` -> 定时任务列表或创建页。
   - `/mcp-web` -> MCP 配置列表或创建页。
   - 路由常量集中维护，避免路径散落在各命令文件中，便于后续与 web-ui 的页面路径同步。

4. 对齐 MCP 与定时任务的字段语义。
   - 复核现有 `timer-commands.ts` 与 `mcp-commands.ts` 的参数定义、提示语、确认摘要。
   - 将 CLI 中的字段标签尽量改成与 Dashboard 表单一致的名称，例如 URL、方法、类型、配置来源等。
   - 保留已有 API 映射逻辑（如 `cron` -> `cron_expr`、`url` -> `callback_url`），但把用户可见层统一成与页面一致的命名。

5. 增加失败兜底与用户提示。
   - 若 Web UI 未配置或未启动，命令应输出明确提示，并展示准备打开的 URL。
   - 若浏览器拉起失败，提示用户手工访问链接。
   - 执行成功后输出“已打开页面 + URL”，保证 CLI 操作可见。

### 接口定义

本任务不新增服务端 API，复用现有 CLI 配置和已存在的后端接口。

| 能力 | 方法 | 路径/来源 | 请求参数 | 响应 |
|------|------|-----------|----------|------|
| 读取 Web 地址 | 本地配置读取 | `config.yaml -> endpoints.webUi` | 无 | Web UI 基础 URL |
| 状态查询 | GET | `/api/v1/health`、`/api/v1/metrics` | 无 | 服务状态与统计 |
| 定时任务创建 | POST | `/api/v1/timers` | `name`、`cron_expr`、`callback_url`、`callback_method` | 定时任务对象 |
| MCP 创建 | POST | `/api/v1/mcps` | `name`、`type`、`command/args/url` | MCP 配置对象 |

- [ ] 在 Windows 环境执行 `/status-web`，浏览器成功打开 Dashboard 状态页。

- [ ] `config.yaml` 中修改 `endpoints.webUi` 后，Web 快捷命令使用新地址。

- [ ] `timer-add`、`mcp-add` 的提示文案与 Dashboard 表单名称保持一致，不影响现有 API 调用。

## 风险与回滚

- 风险 1: Web UI 实际路由可能与 CLI 预设不一致，需在编码前确认 dashboard 页面路径。
- 风险 2: Windows `start` 命令对带查询参数 URL 的转义较敏感，需在实现时单独验证。
- 风险 3: 若直接引入第三方打开浏览器库，会新增依赖并影响打包，应优先尝试零依赖实现。
- 回滚方式: 新增能力主要集中在 REPL 命令注册和工具模块，可通过移除 `web-shortcuts.ts` 注册快速回退，不影响既有 `status`、`timer`、`mcp` API 流程。
