# Check-Todo Skill

## 触发条件
当用户询问"下一步做什么"、"检查 TODO"、或明确调用 `/check-todo` 时触发。

## 执行流程

### Step 1: 读取 TODO 文件
读取 `@docs/roadmap/todo-list/app.js` 中的任务数据，解析当前进度：
- 按 Phase 分组检查（Phase 1 > Phase 2 > Phase 3）
- 优先关注 `in-progress` 状态的任务
- 无进行中的任务时，推荐第一个 `pending` 的任务

### Step 2: 确定下一步任务
根据优先级规则确定下一步：
1. **Phase 1 阻塞项**：如果 Phase 1 未全部完成，优先执行 Phase 1 中的 `pending` 任务
2. **Phase 2 并行项**：Phase 1 完成后，可并行开发 Phase 2 中的任务
3. **Phase 3 扩展项**：最后执行 Phase 3

输出格式：
- 当前 Phase 完成度统计
- 推荐执行的下一个任务（高亮显示）
- 该任务所属模块和关键信息

### Step 3: 标记任务为进行中
在同一会话中，指导用户将选中的任务标记为 `in-progress` 状态：
- 说明需要在浏览器中打开对应的 HTML 页面
- 找到对应任务并点击"开始"按钮

### Step 4: 创建技术方案文件夹
根据任务所属模块，创建标准的技术方案结构：

**目标路径**: `services/{module}/docs/feature/{yyyy-mm-dd-需求名称}/`

其中：
- `{module}`: 从任务中提取（cli/llm/server/web/other）
- `{yyyy-mm-dd}`: 当前日期
- `{需求名称}`: 根据任务标题生成简短的英文或拼音名称

**创建文件**:
根据任务的具体信息（任务标题、模块、notes），**自动生成完整的技术方案内容**，不要留下任何"待补充"占位符。

创建 `技术方案.md`，内容结构如下：

```markdown
# {任务标题} - 技术方案

## 背景与目标
{根据 task.notes 和 task.module 分析得出的背景说明}

## 需求概述
{详细解释这个任务要实现什么功能，基于 task.title 和 task.notes 展开说明}

## 技术方案
### 方案选型
{根据任务所属模块分析：}
- 技术选型: {适合该技术栈的方案}
- 设计模式: {推荐的设计思路}
- 依赖库/工具: {可能需要的第三方依赖}

### 核心实现步骤
{根据任务类型，生成具体的实现步骤，每个步骤要有明确的内容：}
1. {具体步骤1 - 如：初始化项目结构/配置文件}
2. {具体步骤2 - 如：实现核心功能函数}
3. {具体步骤3 - 如：添加错误处理和日志}
4. {具体步骤4 - 如：编写测试用例}

## 接口定义（如需要）
{如果任务涉及 API/接口，用表格形式定义：}

| 接口 | 方法 | 路径 | 请求参数 | 响应 |
|------|------|------|----------|------|
| {接口名} | GET/POST/... | /api/... | {...} | {...} |

## 目录结构（如需要新建文件）
{建议的文件/目录结构：}

{module}/
├── {具体文件或目录结构}
```

## 测试计划
- [ ] {针对该任务的单元测试方案}
- [ ] {集成测试验证点}
- [ ] {手动验证步骤}

## 参考资料
- 相关文档: services/{module}/docs/roadmap/{最新roadmap文档}
- 相关代码: {项目中已有的相关实现路径}


**内容生成规则**：
1. **参考资料**: 必须查找并引用对应模块 `docs/roadmap/` 目录下的最新说明文档
2. **接口定义**: 如果任务涉及 API，必须生成具体的接口表格，不要留空
3. **不要出现以下词汇**: "待补充"、"待填充"、"待调研"、TODO、FIXME
4. 如果某个部分确实不需要（如纯配置任务不需要接口定义），直接省略该章节，而不是写一个"不需要"

### Step 5: 报告并等待
输出以下内容后 **结束执行**：
1. 下一步执行任务的摘要
2. 已创建的技术方案文件路径
3. 提示用户检查并完善技术方案
4. 明确告知："请检查并完善技术方案后，告诉我继续执行"

## 注意事项
1. **不要自动修改** `app.js` 文件中的任务状态（由用户在浏览器中操作）
2. **文件夹命名规范**: 使用小写字母、数字和连字符，避免中文和空格
3. **模块映射**:
   - `cli` -> `services/cli/`
   - `llm` -> `services/llm-py/`
   - `server` -> `services/server-go/`
   - `web` -> `services/web-ui/`
   - `other` -> `docs/feature/` (项目级特性)
4. 如果技术方案文件夹已存在，**跳过创建**并提示用户现有路径

