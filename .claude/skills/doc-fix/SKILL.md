---

name: doc-fix
description: |
全量扫描并更新项目中的 Markdown 文档，用于发版前文档整理与规范化。

使用场景：

* 发版前统一整理文档
* 敏捷开发后补全文档缺失
* 校验文档与代码一致性

---

# 📌 作用

对整个仓库的 Markdown 文档进行一次**全量更新与规范化**，确保：

* 文档与当前代码结构一致
* 缺失内容被补全
* 结构符合规范
* 可用于对外发布

---

# ⚙️ 规则

1. 必须基于真实代码结构更新文档，禁止臆造内容
2. 所有路径、文件、目录必须真实存在
3. 更新时间必须为当前时间（YYYY-MM-DD HH:mm）
4. 文档统一使用中文
5. 保持 Markdown 结构清晰、层级合理
6. 不删除仍然有效的内容（除非确认已废弃）
7. rules 内容避免重复，优先抽象总结
8. README 面向用户，禁止暴露内部规则（如 rules）

---

# 🧭 执行流程

## Step 1：扫描项目结构

* 遍历 `services/*/` 目录
* 获取真实文件结构（目录 + 关键文件）

识别：

* 新增但未记录的文件
* 已删除但仍存在于文档中的内容
* 缺失文档的服务

---

## Step 2：更新服务级 AGENTS.md

对每个 service 执行：

### 2.1 校验文件存在性

* 不存在则创建 AGENTS.md（使用标准结构）

---

### 2.2 更新目录结构

更新 `## 📂 目录结构`：

* 使用 tree 结构展示
* 与真实代码完全一致

要求：

* 补充缺失文件
* 删除无效文件
* 为文件添加简要说明
* 标记新增文件

示例：

```text
services/xxx/
├── src/
│   ├── index.ts        # 入口文件
│   ├── new-file.ts     # ← 新增：xxx功能（YYYY-MM-DD）
```

---

### 2.3 更新时间

更新：

```
🕒 最后更新时间
YYYY-MM-DD HH:mm
```

---

### 2.4 结构校验

确保包含以下章节（缺失则补全）：

* 📌 作用
* ⚙️ 规则
* 🧭 开发流程
* 🚫 禁止
* 📂 目录结构
* 🕒 最后更新时间

---

## Step 3：更新 rules.md

路径：`.claude/rules/rules.md`（或同类文件）

根据代码与开发过程补充：

### 3.1 架构规则

* 模块划分方式
* 目录结构约定
* 依赖关系约束

### 3.2 代码规范

* 命名规范
* 风格约定
* 工具使用方式

### 3.3 常见陷阱（重点）

* 常见报错（如接口 / 类型错误）
* Agent / RAG / CLI 问题
* 易错逻辑

### 3.4 示例参考

* 推荐代码写法
* 优秀实现方式

---

添加格式（必须统一）：

```
规则描述或示例（发现日期：YYYY-MM-DD）
```

---

## Step 4：更新根 AGENTS.md

提供全局视角，必须包含：

1. 项目结构总览（services 列表）
2. 各 service 职责说明
3. 开发流程说明
4. Agent 使用方式（如有）
5. 文档索引导航

---

## Step 5：更新 README.md（发版用）

面向用户，必须包含：

### 核心内容

* 项目简介（清晰定位）
* 核心能力（CLI / Agent / RAG）
* 快速开始（安装 + 使用）
* 示例（命令或效果）
* 架构说明（简略）
* 开源/本地部署说明

### 风格要求

* 简洁清晰
* 偏用户视角
* 不暴露内部规则（如 rules）

---

# 🚫 禁止

* ❌ 编造不存在的文件或结构
* ❌ 修改代码逻辑
* ❌ 删除仍在使用的文档内容
* ❌ 在 README 中暴露 rules 或内部约束
* ❌ 输出与代码不一致的信息

---

# 🎯 输出要求

* 直接修改文件，不输出解释
* 所有文档必须：

  * 结构完整
  * 内容准确
  * 与代码一致

---

# 🧠 使用提示（给 Agent）

当用户执行 `/docfix` 时：

* 自动触发本流程
* 全量扫描并更新所有文档
* 作为“发版前最后一步”执行
