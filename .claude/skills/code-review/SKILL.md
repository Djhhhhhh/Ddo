# CodeReview 技能

自动为服务设置代码审查标准，评估功能变更符合程度，并记录审查结果。

## 执行步骤

### 1. 确定目标服务
根据当前工作目录或指定的服务名称确定要进行代码审查的服务。

### 2. 加载或创建审查标准
- 读取服务的 `.claude/rules/review-standards.md` 文件
- 如果文件不存在，创建默认标准文件
- 标准包括架构、代码规范、最佳实践等维度的评分项

### 3. 分析待审查的变更
- 获取当前git暂存区的变更（或指定的commit范围）
- 解析变更的文件类型、修改范围和影响
- 识别新增、修改和删除的文件

### 4. 应用审查标准评分
- 按照标准文件中的维度对变更进行评分
- 每个维度提供0-5分的评分（0=不符合，5=完全符合）
- 记录不符合项和改进建议

### 5. 生成审查报告
- 计算总分和各维度得分
- 列出发现的问题和建议的改进措施
- 根据总分给出通过/不通过建议

### 6. 记录审查结果
- 将审查结果保存到服务的 `.claude/review-history/` 目录
- 文件命名格式：`YYYY-MM-DD-HH-MM-SS-{commit-short}.md`
- 更新服务的审查统计信息

## 输出格式

### 成功时

```
✓ 代码审查完成！
- 服务: xxx
- 总分: xx/xx
- 各维度得分: 
  - 架构符合性: x/x
  - 代码规范: x/x
  - 最佳实践: x/x
  - ...
- 问题发现: x 个
- 建议改进: x 项
- 审查报告: .claude/review-history/YYYY-MM-DD-HH-MM-SS-{commit-short}.md ✅
```

### 失败时

```
✗ 代码审查失败
原因: [具体错误信息]
```

## 配置选项

- `--service <name>`: 指定要审查的服务名称（默认：当前目录所在服务）
- `--std <file>`: 指定审查标准文件路径（默认：服务的 .claude/rules/review-standards.md）
- `--range <commit-range>`: 指定要审查的commit范围（默认：HEAD）
- `--threshold <score>`: 设置通过阈值分数（默认：80%）
- `--format <format>`: 输出格式（text/json，默认：text）
- `--no-record`: 不记录审查历史（仅用于临时检查）

## 审查标准文件格式

审查标准应存储在 `.claude/rules/review-standards.md` 中，格式如下：

```markdown
# 代码审查标准

## 评分维度

### 架构符合性 (权重: 30%)
- 是否遵循服务边界原则：不跨服务import，只能调用API
- 是否保持模块内聚和低耦合
- 是否遵循分层架构原则

### 代码规范 (权重: 25%)
- 是否遵循语言特定的代码风格指南
- 是否有适当的注释和文档
- 是否使用了有意义的命名

### 最佳实践 (权重: 20%)
- 是否处理了错误和边界情况
- 是否避免了常见的反模式
- 是否考虑了性能影响

### 测试覆盖 (权重: 15%)
- 新功能是否有对应的测试
- 是否保持或提高了现有测试覆盖率
- 单元测试是否覆盖了关键路径

### 文档完整性 (权重: 10%)
- 是否更新了相关文档
- 是否有适当的API文档
- 是否说明了变更的原因和影响

## 不通过条件
任意维度得分低于2分视为不通过该维度
```

## 审查历史文件格式

每次审查生成的历史文件格式如下：

```markdown
# 代码审查报告

**服务**: xxx
**提交**: abc1234 (feature/add-user-login)
**日期**: 2026-04-14
**审查者**: AI Code Reviewer

## 总体评分
- 总分: 85/100
- 通过状态: ✅ 通过 (阈值: 80)

## 各维度得分
- 架构符合性: 25/30
- 代码规范: 20/25
- 最佳实践: 15/20
- 测试覆盖: 15/15
- 文档完整性: 10/10

## 发现的问题
1. 架构符合性: 在utils.py中直接引用了server-go的内部函数
   - 建议: 通过API调用而不是直接import
2. 代码规范: 变量命名不够描述性
   - 建议: 使用更具语义的变量名称

## 改进建议
1. 重构utils.py以通过API调用server-go服务
2. 统一变量命名风格，遵循PEP8标准
3. 为新增功能添加单元测试覆盖

## 变更统计
- 文件变更: 3个新增, 2个修改, 0个删除
- 代码行数: +120 / -15
```

## 注意事项

1. 审查标准应根据服务的具体情况进行定制
2. 建议在功能开发完成后、提交前运行此技能
3. 审查结果应作为代码提交的参考，但不是绝对的禁止条件
4. 定期回顾和更新审查标准以适应技术发展
5. 可以将此技能集成到CI/CD流水线中作为质量门禁

## 扩展点

- 支持自定义审查维度和权重
- 支持不同编程语言的特定规则
- 支持与外部代码审查工具集成
- 支持生成趋势报告和仪表盘
