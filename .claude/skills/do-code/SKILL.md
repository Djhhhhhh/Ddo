# Do-Code Skill

## 触发条件

1. 用户明确调用 `/do-code`
2. 执行完 `/check-todo` 后，用户确认"继续执行"自动触发

## 执行流程

### Step 1: 定位技术方案

根据以下优先级找到技术方案：

1. **参数指定**：`--feature <path>` 直接指定技术方案目录
2. **自动查找**：检查当前服务 `services/{module}/docs/feature/` 下最新的技术方案目录（按日期排序）
3. **用户确认**：如果找到多个，列出让用户选择

读取技术方案文件 `技术方案.md`，提取关键信息：
- 技术方案标题
- 目录结构（要创建的文件/目录）
- 核心实现步骤
- 接口定义
- 测试计划

### Step 2: 实施编码

根据技术方案实施编码：

1. **创建目录结构**
   - 按照技术方案的目录结构创建所有必要目录
   - 确保文件路径正确

2. **实现功能代码**
   - 按照核心实现步骤逐个实现
   - 编写符合服务规范的代码
   - 确保错误处理和边界条件
   - 参考服务的 rules.md 遵循架构规则

3. **编写配置文件**
   - 如有必要，更新 package.json、tsconfig.json 等

**编码原则**：
- 优先完成核心功能，确保可运行
- 不要过度设计，按技术方案实现即可
- 保持代码简洁、可读
- 遵循服务已有的代码风格

### Step 3: 更新 AGENTS.md

编码完成后，更新服务的 `AGENTS.md`：

1. **更新目录结构章节**
   - 在 `## 📂 目录结构` 下添加新创建的文件/目录
   - 使用 tree 格式展示
   - 为每个新增文件添加简要注释

2. **更新最后更新时间**
   - 修改 `## 🕒 最后更新时间` 为当前时间

**示例**：
```markdown
## 📂 目录结构

services/cli/
├── src/
│   ├── index.ts              # CLI 入口，命令注册和解析
│   ├── commands/
│   │   ├── init.ts           # init 命令实现
│   │   └── new-feature.ts    # ← 新增：xxx 功能（2026-04-14）
```

### Step 4: 更新 rules.md

编码过程中或完成后，如发现以下情况，更新 `.claude/rules/rules.md`：

1. **新增架构模式**
   - 在 `## 架构规则` 下添加新的设计模式、包结构规则

2. **新增代码规范**
   - 在 `## 代码规范` 下添加新的编码约定

3. **发现常见陷阱**
   - 在 `## 常见陷阱` 下记录遇到的问题和解决方案

4. **添加示例参考**
   - 在 `## 示例参考` 下添加好的代码示例

**添加格式**：
```markdown
- 规则描述或代码示例（发现日期：YYYY-MM-DD）
```

### Step 5: 创建功能测试清单

在对应的技术方案目录下创建 `review-list.md`：

**文件路径**：`services/{module}/docs/feature/{yyyy-mm-dd-需求名称}/review-list.md`

**内容结构**：
```markdown
# {需求名称} - 功能测试清单

**创建时间**: YYYY-MM-DD HH:MM:SS  
**技术方案**: [技术方案.md](技术方案.md)

## 测试覆盖范围

|--------|------|--------|------|

### {功能模块1}
- [ ] {验证点1} - 预期结果: {xxx}
- [ ] {验证点2} - 预期结果: {xxx}
```

**测试清单生成规则**：
1. 根据技术方案的"测试计划"章节提取测试项
2. 根据"接口定义"章节生成功能验证点
3. 根据"核心实现步骤"生成边界情况测试
4. 优先级：核心功能 > 边界情况 > 回归测试
5. 针对api的测试要给出对应的curl

## 配置选项

- `--feature <path>`: 指定技术方案目录路径
- `--service <name>`: 指定目标服务（自动检测时可省略）
- `--dry-run`: 预览模式，显示将要创建/修改的文件但不实际执行
- `--skip-docs`: 跳过更新 AGENTS.md 和 rules.md
- `--skip-review`: 跳过创建 review-list.md

## 注意事项

1. **编码前先阅读和确认**技术方案的完整内容
2. **保持最小化修改**：只创建技术方案中明确的文件
3. **文档更新是强制的**：除非使用 `--skip-docs`
4. **review-list 必须包含**：功能点、边界情况、回归测试
5. **如有编译错误**：尝试修复，如果超出技术方案范围则记录到 review-list 的"已知问题"

## 示例

### 基本使用

```
/do-code
```

自动查找最新技术方案并执行编码。

### 指定技术方案

```
/do-code --feature services/cli/docs/feature/2026-04-14-ddo-init
```

### 预览模式

```
/do-code --dry-run
```

显示将要创建/修改的文件列表，但不实际执行。
